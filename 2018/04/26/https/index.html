https协议传输过程

一、HTTP协议与HTTPS协议

HTTP协议具有以下不足：

  • 通信使用明文，内容可能被窃听；
  • 无法验证报文的完整性，可能遭遇篡改；
  • 不验证通信双方的身份，可能遭遇伪装。
    HTTPS协议：
    CIA
  • confidential:隐秘的，你浏览的页面内容如果被人从中间截获，将会是一团乱码；；
  • integrity:完整，你浏览的页面就是原本想要浏览的，服务器收到的数据也是你最初发送的。数据传输不会被黑客篡改或者替换掉；
  • authentication: 认证，连接的网站不会是中途被人伪造的。

HTTP+加密+认证+完整性保护=HTTPS

SSL建立在可靠传输协议（如TCP）之上，为高层协议提供数据封装、压缩、加密等基本功能的支持。
SSL握手协议(SSL Handshake Protocal):用于在实际的数据传输开始时，对通信双方进行身份认证、协商加密算法、交换加密秘钥等。

2、秘钥协商过程-TLS握手

2-1 客户端发出请求(Client Hello)

客户端主要向服务器提供以下信息：
（1）支持的协议版本——Version:TLS 1.2
最后使用什么样的版本，得由服务端决定。如果服务端不支持的话，客户端得降版本。
（2）一个客户端生成的随机数——Random
最后加密数据用的主密钥，需要客户端和服务端一起协商出来。后面服务端的 Server Hello 阶段也会生成一个随机数,一同用来计算出主密钥。
（3）会话ID——Session ID
这个 Session ID 是可以重用的，具体看服务端资源和支持情况。如果要复用 Session ID， SSL 服务端需要维护连接的状态和上次握手成功留下的加密信息。如果客户端保存了 Session ID 的信息，发起 SSL 请求的时候会带上。
（4）加密套件——Cipher Suites
是一个列表，里面每条记录都包含：秘钥交换算法、加密算法等。
秘钥交换算法是非对称加密，用于在SSL握手阶段交换协商好的对称秘钥；
加密算法是对称加密，用于后面加密HTTP数据；
摘要算法对数据进行摘要，后面可以用于对数据进行校验，保证数据一致性，防止中途被篡改。
总结：用非对称加密算法来传递对称加密算法的秘钥，同时用摘要算法保证数据完整性。
（5）压缩算法——Compression Methods
（6）扩展字段——Extension

2-2 服务端回应

(Server Hello)
服务端收到客户端的Client Hello后，会反馈自己的配置信息，包括：
（1）协议版本——指定此次SSL使用TLS 1.2
（2）随机数——Random
（3）会话ID——Session ID
与上次Client Hello中的sid不一样，说明会话已过期，服务端重新创建了一个会话。
（4）加密套件——Cypher Suite
从上次的Client Hello 包中的Cipher Suites中选择了一条TLS_ECDHE_RSA_WITH_AES_128_CBC_SHA。
即交换加密算法为ECDHE_RSA，加密算法为AES，摘要算法为SHA。

(Certificate)
服务端下发证书，客户端验证服务端身份，并取出证书携带的公钥，这个公钥是交换加密算法的公钥，也就是Server Hello阶段指定的ECDHE_RSA。

(Client Key Exchange)
客户端不直接生成加密密钥，而是通过之前客户端和服务端生成的随机数又再生成一个随机数，使用前面协商好的EC Diffie-Hellman 算法进行加密传输给服务端。
在服务端收到了 Client Key Exchange 消息后，会使用自己的私钥解开这个随机数，加密密钥交换完成。
(Change Cipher Spec)
客户端通知服务端，后续的报文将会被加密。
